## 1. 问题清单（已核实代码）

### 问题 A — 牌桌内无"开始游戏"按钮 🔴 前端缺失

**现象**: 房主创建房间进入牌桌后，没有按钮开始这一局。

**根因**: 全前端只有 `GameSelection.tsx`（主页）有"开始游戏"按钮，用途是"进大厅"。牌桌页 `TablePage.tsx` 没有触发 `table:start_hand` 事件的按钮。

**契约**: `table:start_hand` 事件已定义（API-CONTRACT §2.4），后端 `@sio.on('table:start_hand')` 已实现，仅前端缺触发入口。

### 问题 B — 创建房间座位逻辑矛盾 🔴 前端逻辑错

**现象**: 创建 6 人桌时，第 3 步 AI 配置列出 0~5 全部 6 个座位让用户配 bot，用户困惑"6 个都能配 bot，那我自己不参与吗？"

**根因**: 
- 后端逻辑正确：房主自动入座 **0 号位**（sio.py:233），bot 占其他座位。
- 前端错误：`CreateTableModal.tsx:240` 用 `Array.from({ length: seats })` 从 i=0 渲染，把房主自己的 0 号位也列进了"配 bot"列表。

**应改为**: AI 配置只列**座位 1 ~ (seats-1)**，0 号位标注为"你（房主）"，不可配 bot。

### 问题 C — 死局残留 🟡 测试数据未清理

**现象**: 登录后看到多个"进行中"的人机局（德州E2E、炸金花E2E、掼蛋E2E、4人掼蛋），是 e2e_harness 测试脚本留下的，无真人在玩。

**根因**: e2e 测试创建的房间未清理，常驻内存。

**应提供**: 死局清理机制（手动接口 + 可选自动回收）。

## 2. 需求详述

### 2.1 牌桌"开始游戏"按钮（问题 A）

**位置**: 牌桌页面（TablePage / TableShell），房主可见。

**显示条件**:
- 当前用户是房主（座位 0）
- 牌局未开始（stage 为初始态 / waiting）
- 满足最小开局人数（德州 ≥2、炸金花 ≥2、掼蛋 =4）

**交互**:
- 点击 → emit `table:start_hand` { table_id }
- 开局后按钮消失（牌局进行中不显示）
- 人数不足时按钮禁用 + 提示"需至少 N 人"

**非房主**: 看到"等待房主开始"提示，无按钮。

### 2.2 创建房间座位修正（问题 B）

**CreateTableModal 第 3 步（AI 配置）改动**:
- 座位 0 固定标注 **"你（房主）"**，不可选 bot，不出现在配置列表
- 只渲染座位 **1 ~ (seats-1)**，每个可选：无 / easy / normal
- 右侧预览：`1 真人（你）+ N 个 AI + M 个空位`

**示例（6 人桌）**:
```
座位 0: 你（房主）        ← 固定，不可配
座位 1: ○无 ○简单 ○进阶
座位 2: ○无 ○简单 ○进阶
座位 3: ○无 ○简单 ○进阶
座位 4: ○无 ○简单 ○进阶
座位 5: ○无 ○简单 ○进阶
```

**掼蛋特殊**: 固定 4 人（2v2），座位 0 是你，1/2/3 可配 bot（但需保证能凑齐 4 人才能开局）。

### 2.3 死局清理（问题 C）

**方案（后端）**:

1. **手动清理接口**（开发/管理用）:
   - `DELETE /api/lobby/tables/{table_id}` — 删除指定房间
   - 或 `POST /api/lobby/cleanup` — 清理所有无真人的房间

2. **自动回收**（可选，推荐）:
   - 房间内所有真人离线超过 N 分钟 → 自动销毁
   - 或纯 bot 房间（无真人）创建后 N 分钟无真人加入 → 销毁

**v1 最小**: 先做手动 `POST /api/lobby/cleanup` 清理无真人房间，够用。

## 5. 技术注意

- **开局按钮的房主判定**: 用 `table:state` 里 your_seat==0 或 players[0].sid==自己
- **最小人数**: 各引擎有 `can_start` / `min_players`，前端按钮禁用条件应对齐后端校验
- **cleanup 安全性**: 只删 `players` 里无真人（全是 bot 或空）的房间，正在玩的真人局绝不能误删
- **掼蛋必须 4 人**: 创建时若 0 号位+bot 不足 4 人，开局会失败，前端应提示补满
